## 三、❌ 文档声称，但实际**未实现 / 桩 / 不通**（重点）

### 1. 🔴 沙盒 sidecar 的对象存储端点完全缺失 → 连累 MinerU OCR 解析 + 工作区归档

- **证据**：`sandbox-service/app.py` **只有 5 个路由**：`/sessions`、`/exec`、`/files`、`DELETE /sessions/{id}`、`/healthz`。全文**无 `/storage/put`、`/storage/delete`，无 boto3/oss2/presign**。
- 但 Go 侧 `internal/storage/storage.go` 明确 POST 到 sidecar 的 `/storage/put`（:88）/`/storage/delete`（:103）。**契约的 sidecar 一半从未实现**。
- **后果一（§4.11-C MinerU）**：非纯文本里**需要 OCR 的扫描件 / 图片 / 老格式**文档走 MinerU，前提 `sb.Put()` 上传到桶拿预签名 URL（`parser.go:322`）→ 打到不存在的 `/storage/put` → 失败；未配存储时直接落 placeholder（`parser.go:139` 一行"binary document…"）。→ **扫描件/图片类文档的 OCR 解析功能不通**。（**文本层 PDF/DOCX/PPTX 走本地抽取，不受影响、可用**。）
- **后果二（§4.5 工作区归档）**：sidecar `_reaper()`（app.py:649）30 分钟回收只是**杀容器**,无 gzip→S3 归档、无再次执行恢复。文档说的"/workspace 归档对象存储、按需恢复"**未实现**,实际是"reaped = gone"（即文档自己列的兜底分支）。
- **修复方向**：在 sidecar 实现 `/storage/put`、`/storage/delete`（boto3/oss2，按 `storage` 块懒加载，prefix 校验）+ reaper 归档/恢复;或在 Go 侧改为直连 S3/OSS（放弃"Go 零云 SDK"原则）。

### 2. 🟠 §2.4 的 Redis "卸压缓存层"基本未落地

- **证据**：实际用到缓存的只有**运营类** key：`verify:`/`reset:`（验证码）、`oauth:state:`、`conv:{id}:stop` 与 `user:{id}:kill`（pub/sub）、`2fa:`、`rl:`（限流）、`quota:v2:`（配额）、`gen:active:`（并发）。
- **未实现**：`cfg:channels`/`cfg:models`（配置缓存）、`user:{id}:settings`、`conv:{id}:recent`（会话热数据）、`emb:{sha256}`（查询向量缓存）、`rag:{sha}`（检索结果缓存）、`user:{uid}:mem`（记忆缓存）、以及**配置变更的 Pub/Sub 失效广播**。
- 即：**渠道/模型/设置/记忆每次都直接查 DB**（`store.GetSetting` 等命中 DB），没有文档承诺的"每条消息不打 PG"的缓存。
- **影响**：功能正确(读 DB 即可)，但 §2.4 宣称的"给 PG/模型 API 卸压"在当前代码里**几乎不存在**;高并发下会成为热点。属规模化优化,非功能缺失。

### 3. 🟠 厂商 Files API 上传未实现（用 base64 内联替代）

- **证据**：`files` 表有 `provider_file_refs` 列，但 provider 代码里**无任何 `/v1/files`、Files API、file_id/fileUri 上传**。视觉走 base64 内联、数据文件走沙箱 `PutFile`。
- 文档 §5/§4.10/§2.3-E 的"Files API（file_id/fileUri）/ provider_file_refs 按厂商存"**未实现**——DESIGN §4.6 的"历史注"其实已承认改为本地磁盘 + base64 内联。
- **影响**：大文件视觉/超大附件走 base64 会更占 token；功能上图片/PDF 视觉可用。

### 4. 🔵 沙盒 `GetFile` 是死桩

- **证据**：`sandbox/sandbox.go:235 GetFile` POST `/files/get`，但 sidecar **无该路由** → 调用必 404。当前无业务路径触发它，属 Go↔sidecar 契约漂移的遗留死代码。

### 5. 🔵 Redis 协调类：幂等键 / 分布式锁 未实现

- 文档 §2.4 的 `idem:{key}`（幂等）、`lock:sandbox:{conv}`/`lock:compact:{conv}`（分布式锁）在代码中**无对应实现**（单实例下并发建沙箱/重复压缩靠进程内顺序，未加锁）。

### 6. 🔵 可选 P2 项（文档多已标 P2/可选，未实现）

- RAG **rerank**（Voyage rerank-2）：未实现（只到 RRF 融合）。
- **语义切块**（按相邻句相似度断）：未实现（递归边界切已实现）。
- 记忆**向量检索**（Qdrant memories collection 按 user 过滤）：未实现（记忆全量注入，文档也说"少时可暂不建向量"）。
- 图片 `image_ref`→原图回显：文档 §4.11-C 自述"本版本未做"。

## 五、结论清单（按"会不会影响用户实际功能"排序）

| 严重度 | 项 | 用户可感影响 |

| ✅ 诚实标注 | §11 分布式整章为未来规划 | 文档已声明,非缺陷 |

> 核心判断：**没有"假装实现/mock"的核心功能**——模型调用、工具、对话/RAG（文本）、记忆、技能、项目、对话树、压缩、计费、鉴权都是真代码真跑通。最该补的是**沙盒 sidecar 的对象存储端点**（一处缺失同时卡住 MinerU OCR 与工作区归档两个文档明确声称的能力）。
