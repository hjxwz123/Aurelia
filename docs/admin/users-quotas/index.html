Users, quota and credits

Blocking, Unlocking and Deleting​

  • ** Banned ** is globally effective in real time: old tokens immediately expire, ongoing generations are discontinued, refresh tokens are cancelled, mechanisms are detailed Overall Background Unlock the button in the same location.
  • ** Eliminated ** The user has a confirmation window and cannot be revoked. delete is ** Asymmetrically ** Confirm that the account is immediately locked out (with the blocking mechanism), the "deleted" logo appears in the list, reset (conversation, message, file, vector, disk cleaning) is completed by the background task batch, and the line automatically disappears after completion - the deletion of the large account no longer holds the page. the deleted account can not be unlocked "reset"; if the process starts again, the unfinished deletion will continue automatically.
  • Self-protection: You cannot block or remove yourself.

Model of quota​

在** Editing page model ** The “Permissions” partition (model > model > settings) configures chat and image model by group:

  • ** granted switch. ** Open indicates that the group can use this model. ** Once a group is assigned, the unallocated group cannot see/use the model. ** At all shutdown, the model is open to all unlimited.
  • ** The Cycle ** A fixed window per N days or N hours.
  • ** Type of upper limit ** Two choices one.
    • ** number of times ** How many times can the window be called?
    • ** Quantity ** The upper limit of the cumulative cost per model quota in the window.
  • ** The upper limit **:0 indicates there is no limitation during the cycle.

** Acts beyond the free quota **: If the credits system is enabled (credits_per_usd greater than 0) and the user has a balance, the model turns to that user ** Credit Deposit by Real Cost ** Continued use; credits are not sufficient, or when the credits system is not enabled, requests are intercepted and the "excess reminder" text is displayed.Administrator accounts and "no quota line model" are not subject to this limit.

Free + credits combination

Typical configuration: to a Free group model "20 times every 7 days", to the group "per cycle credits" send some time-effective credits. users first consume the number of free times, after using automatically turned into deductible credits, the two quotas independent of each other.

exchange and exhibition.​

  • Deposit amount = Cost calculated by model quota for the call (USD) ×credits_per_usd。
  • model picker for walking credits conversation model display ** Relative multiplier logo ** ×N, N = (input price + output price) ÷ 5, retaining a small number, allowing the user to intuitively compare the cost of each model.
  • On the billed image model, when the free exhaust is converted to deductible credits, the selector displays "N credits" after the model name, N = single image price ×credits_per_usd (Example: one $0.2 credits_per_usd= 100 to show 20 credits).
  • Users can see the current credits balance (including the expiration time) and the permanent credits balance on the subscription page; each AI response also records the consumption of this round.

Payment time: send out immediately, failure is not counted​

  • ** Normal completed **: Count the number of quota times in the window, deduct credits by full amount.
  • ** The user stops. ** As long as the model has produced content, the same count and deduct credits according to the actual cost of the token produced.
  • ** Failure or interception before sending ** (upstream reporting error, quota excess, moderation interception): no counting, no deduction.

Pre-examination of credits​

Forecasting and blocking insufficient balance requests (open by default) in System > Global Settings: ** Go to Credits ** The round, after the real request assembly is completed, before it is sent to the model, predicts the input token (plus a fixed 2k output reservation) and exchanges it for credits; if the user's available balance (time effect + permanent) is exceeded, it is directly rejected by "insufficient balance" and does not generate any upstream consumption.

Management and State​

  • ** Filtered ** Filter by status (all / unused / converted / disabled / expired) and batch names; list 20 items per page.
  • ** State of ** Unused, partially used (partly used in multiple codes), exchanged (full), disabled, expired.
  • ** Prohibited / Activated ** Disable reversible, retain audit records, and unchanged codes immediately expire.
  • ** Eliminated ** Remove the single code or the entire batch, together with the exchange record removed, can not be revoked; ** Members already awarded are not affected. ** The end period naturally returns.
  • ** Exchange of records ** Each code can see the exchange audit (who, when, from which group to switch).

Exchange behavior and limits​

  • exchange for the user. ** Changed immediately ** If the group of code is different from the user's current group, the server first returns the confirmation request, the front-end pop-up message "will be switched to the X group" comes into effect after the user's confirmation.
  • It automatically returns to the user group before the exchange.
  • ** Explosion restrictions ** Switch the interface by IP stream limit, by default up to 10 times within 60 seconds, environmental variables available AIVORY_API_RATE_LIMIT_REDEEM_CODE_MAX / AIVORY_API_RATE_LIMIT_REDEEM_CODE_WINDOW adjusted to see. Advanced environmental variables。
redeem code does not directly fill credits

The current version of the redeem code is ** User group + length **; credits are issued through the user group's "per-cycle credits", and permanent credits are manually adjusted by the administrator on the user's editing page.
