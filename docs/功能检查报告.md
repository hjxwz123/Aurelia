## 二、按模块详细报告

### A. 认证与账户安全

**A1 — OAuth 未验证邮箱联结既有账户（账户接管）** 🔴
- 现状/证据：`resolveOAuthUser`（`oauth_handlers.go:191-207`）在「已验证邮箱」分支后，对**未验证**邮箱仍 `FindUserByEmail` 命中即 `LinkOAuthIdentity` 联结并返回该账户（`:204-207`，注释明确「unverified collision: link rather than fail」）。叠加 `decodeIDToken` 不验签（`oauth.go:319`），一个 `kind=oidc`（管理员可加）或返回未验证邮箱的 IdP 可断言任意 `email` 接管同名本地账户。
- 修复：未验证邮箱命中既有账户时**拒绝**（或走显式「已登录态下绑定」确认）；仅当邮箱不冲突时才自动建号；对 oidc 校验 id_token 的 JWKS 签名 + `iss/aud/exp` + 要求 `email_verified`。

**A13 — 非 production 默认 JWT 密钥 / 默认管理员密码** 🔴(误部署)
- 证据：`defaultDevJWTSecret="dev-secret-change-me-aurelia-2026"`、`SeedAdminPass` 默认 `"aurelia-admin"`（`config.go:11,65`），`Validate` 仅在 `Env==production|prod` 时拒绝（`config.go:88-99`），而 `AURELIA_ENV` 默认 `development`。忘记设 `AURELIA_ENV=production` 即用**公开已知**的 HS256 密钥启动 → 任何人可伪造 `{role:"admin"}` token。
- 修复：任何非 dev 环境一律 fail-closed 拒绝默认密钥；监听非回环地址时拒绝默认密钥；启动打印醒目告警。

**A2 — refresh token 不轮换/不撤销** 🟠
- 证据：`refreshHandler`（`auth_handlers.go:308`）签发新 jti 但**不撤销**所出示的旧 jti；`IsRefreshTokenValid`（`misc.go:340`）仅查存在/未撤销/未过期。捕获一次的 refresh 可在 30 天内反复使用并不断派生新 token，无重放检测。
- 修复：每次刷新撤销出示的 jti（旋转），复用已撤销 jti 视为入侵信号 → 撤销该用户全部 token + bump token_ver。

**A3 — 无 CSRF 防护** 🟠
- 证据：`requireAuth` 接受 `auth_token` **Cookie**（`middleware.go:81`），所有变更类路由仅靠 Cookie + `SameSite=Lax`；无 CSRF token、无 Origin/Referer 校验；`x-aurelia-csrf` 仅在 CORS 头里出现（`router.go`）但从未被读取，前端也不发。
- 修复：对所有非 GET 路由加双提交 CSRF token 或严格 `Origin`/`Sec-Fetch-Site` 白名单；或变更类操作只认 `Authorization: Bearer`（跨站攻击者无法设置该头）。

**A4 — 改密/重置不撤销 refresh** 🟠
- 证据：`UpdateUserPassword`（`users.go:205`）只 `BumpTokenVersion`（杀 access），不改 `refresh_tokens.revoked`；`resetPasswordHandler`/`changePasswordHandler` 均依赖它。持有被盗 refresh 的攻击者在受害者改密后仍可 `/refresh` 重建会话，重置失去意义。
- 修复：`UpdateUserPassword` 内一并 `UPDATE refresh_tokens SET revoked=1 WHERE user_id=?`。

**其他（MEDIUM/LOW）**
- 🟡 A5 验证码/2FA 无每账户/每码节流（仅每 IP+路径）；6 位重置码 10min 有效、`login2faHandler` 错码不销毁 ticket（`twofa_handlers.go:128`）；`logout`、`oauth callback` 无限流。修复：每邮箱/每 ticket 失败计数，N 次后销毁。
- 🟡 A7 id_token 未验签（同 A1）。
- 🟡 A10 2FA ticket 经 `/login?twofa_ticket=` URL 传递（`oauth_handlers.go:171`）→ 可入浏览器历史/Referer/日志。修复：改用一次性 POST/会话绑定。
- 🔵 Cookie `Secure` 硬编码 true（`auth_handlers.go:383`）→ 纯 HTTP dev 丢 Cookie。🔵 登录可枚举账户状态（pending/banned 在校验密码前即区分，`auth_handlers.go:263-276`）。🔵 注销账户仅密码确认、未要 2FA 码，且未清理 `files/shares/kbs/artifacts`（`users.go:298`）。🔵 TOTP secret 明文存储（`schema.sql:25`）。🔵 `adminDisableTwofaHandler` 不校验目标存在。

### B. 模型编排 / Provider / 配额 / 记忆

**B1 — 提示注入驱动工具** 🟠
- 证据：唯一防御是系统提示「信任边界」文字（`orchestrator.go:919`）与块包裹标签；工具执行层无门禁（`orchestrator.go:1279`→`tools/registry.go:81`）。被投毒的 KB 文档或 `web_fetch` 抓取页可指示模型 `web_fetch` 攻击者 URL（外泄会话内容）、刷 `image_generate`（烧钱）、跑任意 `python_execute`。深度研究每轮自动抓取最多 25 个攻击者可控页喂给写作模型。
- 修复：非搜索结果 URL 的 `web_fetch` 需用户/策略显式许可；源自 `<tool-output>` 的工具调用降权，禁止仅凭工具输出触发 `python_execute`/`image_generate`。

**B3 — 花费配额竞态** 🟠
- 证据：cost 型 `recordQuotaUsage` 做 `Get→parse→add→Set`（`quota.go:75-79`），跨读改写无锁/CAS；内存 cache 无原子浮点加（`cache.go`）。允许 3 并发生成时交错丢增量 → 超额。窗口边界还会把跨界轮成本写进新窗口键。
- 修复：花费用整数微单位 + 原子 `IncrBy`，或对 key 加锁/Redis Lua。

**B4 — 配额仅生成前校验** 🟠
- 证据：`checkModelQuota` 仅生成前一次（`orchestrator.go:256`），计数生成后才写（`:636`）。卡在 99% 的用户可发起一次深度研究（4 轮×6 搜+4 抓 + 全文写作 + plan/verify），单轮无中途上限。
- 修复：加硬性每轮花费上限 + 深度研究独立预算，按 token/证据累积增量执行，超限即中断。

**B5 — 上游错误回传 + Google key 在 URL** 🟠
- 证据：各 Provider 把 `fmt.Errorf("<vendor> %d: %s", status, body)` 原文经 SSE `error` 事件写到线上（`messages_handlers.go:118`）并存入 `messages.error` 再回传所有者；Google 把 key 放查询串 `...streamGenerateContent?...&key=%s`（`google_provider.go:86`）→ 任何代理/访问日志泄露密钥。
- 修复：用户侧返回通用错误、详情仅服务端日志；Google key 改走 `x-goog-api-key` 头。

**其他（MEDIUM/LOW）**
- 🟡 B2 `retry.go` 死代码（无 429/5xx 韧性），`http.DefaultClient` 无超时。
- 🟡 B6 `registry.List(_)` 忽略 modelID → 工具集不可按模型细分（与 F10 同源）。
- 🟡 B7 记忆投毒：抽取不区分内容来源是否为用户（`memory_worker.go:90`），可把工具输出里的「指令」升格为系统提示里的 `[CURRENT]` 事实。修复：只从 `role==user` 文本抽取。
- 🟡 B11 配额/每日限制在 DB 错误时一律 fail-open（`quota.go:33`、`builtins.go:664`、`middleware.go:257`）。
- 🔵 `computeCost` 用 `OutputTokens` 兼作图片数、token 数无上界（`orchestrator.go:1167`）；🔵 `messages.raw`/`messages.error` 未脱敏下发所有者（`redactCost` 只清 cost）；🔵 提示模式工具失败立即重试 3 次放大副作用（`prompt_tools.go:219`）；🔵 `tallyTurnSideCosts` 60s 兜底未实现（注释与实现不符）。

### C. 知识库 / 文档 / RAG / 上传

**C1 — 跨用户 KB/文档读取（已人工复核）** 🔴
- 现状/证据：`updateConversationHandler` 把 `kb_ids` 原样写入会话且无所有权校验（`conversations.go:208`）。编排器把 `conv.KBIDs` 直接并入检索作用域（`orchestrator.go:285-289`），调用 `rag.Retrieve(ctx, conv.UserID, conv.ID, kbIDs, ...)`。但 `Retrieve` 中 `userID` **仅用于记账**（`rag.go:345`），真正的检索作用域无论 Qdrant（`rag.go:364/368`）还是关系库 `ListChunksInScope`（WHERE `kb_id IN(...) OR conversation_id=?`）**都不含 user_id**。→ 用户把自己会话的 `kb_ids` 设为他人 KB id 即可读出其文档分块。唯一门槛是获知 48-bit 的目标 KB id（仅所有者可列出，但可能经 URL/截图/日志泄露）。
- 修复：编排器与 `updateConversationHandler` 对每个 `kb_id` 走 `store.GetKB(id, conv.UserID)` 校验并剔除非自有；向量 payload + 分块查询补 `user_id` 谓词做防御纵深。

**C2 — 向量/分块作用域无 user_id** 🟠：根因同 C1，`qdrant.go:178`/`kbs.go:284` 过滤只有 kb_id/conversation_id。修复：scope 内贯穿 `userID`。

**C3 — 上传无大小上限** 🟠：`uploadFileHandler`/`receiveDocument` 用 `ParseMultipartForm` 后 `io.Copy` 无 `LimitReader`（`files_handlers.go:107,160`），`MaxUploadBytes` 实际未在流上强制（对比 icon/audio 正确用了 `io.LimitReader`）。修复：`http.MaxBytesReader` + `io.LimitReader(file, max+1)` 超限删档。

**C4 — 上传/ingest 无限流** 🟠：仅 `/auth/*` 有限流，上传路由只有 `requireAuth`（`router.go:122,128`）；每次上传触发付费 MinerU OCR + 计费嵌入，单用户可无限提交大 PDF。修复：每用户固定窗口上传限 + 每日 ingest/嵌入 token 预算。

**C5 — 提升文档不重嵌入** 🟠：`PromoteDocumentToKB` 仅 `UPDATE chunks SET kb_id=?`，会话文档用全局（可能 256 维本地哈希）嵌入，目标 KB 锁定别的维度 → 提升后检索静默失效。修复：提升时按目标 KB 锁定 embedder 重嵌入重 upsert，或维度不符时拒绝。

**其他（MEDIUM/LOW）**
- 🟡 C6 MinerU `full_zip_url` 被服务端直取无 host 白名单（`parser.go:491`）→ SSRF（需 MinerU 响应可控）。
- 🟡 C7 `regenerateHandler` 漏 `checkDailyMessageLimit`（`messages_handlers.go:133`）→ 反复 regenerate 绕过每日消息上限。
- 🟡 C11 前端流中断不调 `reloadActivePath` 对账（`conversations.ts:549`，与 E7 同源）。
- 🔵 编辑/发送无服务端文本长度上限；🔵 就地编辑遗留旧助手回答（`conversations.go:441`）；🔵 上传 MIME 由客户端控制并存储；🔵 Office/PDF 本地提取无聚合大小/条目上限；🔵 解析失败把上游错误原文写入文档 `error` 回显；🔵 fork 不复制会话级文档。
- ✅ 已正确：artifact 下载按所有者 + 路径前缀校验（`files_handlers.go:262`）；zip-slip 防护 `zipNameIsSafe`（`parser.go:603`）；upload_policy 扩展名白名单。

### D. 管理后台 / 用户组 / 订阅 / 分享

**D1 — 分享 token 可枚举 + 公开端点无限流** 🟡
- 证据：token = `genID("sh")` = `sh_` + 12 hex = **48-bit**（`shares.go:33`），`GetShareByToken` 无 scope（`shares.go:67`），`GET /api/public/shared/:token` 无 `rateLimitedIP`（`router.go:62`）。可暴力枚举他人分享快照（含完整对话文本）。
- 修复：token 用独立 ≥128-bit `crypto/rand`；公开分享端点加限流。

**D2 — 管理员自封禁 / 末位管理员无保护** 🟡
- 证据：`banUserAdmin`（`admin_handlers.go:268`）无 `id==self` 校验、无末位管理员校验（`setUserRoleAdmin` 有自我校验但 ban 没有）；`Seed` 仅在「无任何 admin」时重建（`store.go:117`）→ 被封但仍存在的 admin 不触发重建，平台可被锁死。
- 修复：禁止自封禁；封禁/降级 admin 前断言至少剩一名活跃 admin。

**其他（MEDIUM/LOW）**
- 🟡 D6 语音代理无限流 + 全量入内存（`audio_handlers.go:52`），可耗内存/烧转写额度。修复：每用户限流/并发 + 流式转发。

### F. 代码沙盒与工具（最高风险）

**F1 — sidecar root + docker.sock = 宿主机接管面** 🔴
- 证据：`docker-compose.yml:39` 挂载 `/var/run/docker.sock`；`Dockerfile.sidecar` 未设 `USER`（root）；`app.py` 用 Docker CLI 起容器。sidecar 对宿主 Docker 有完全控制；无 userns-remap、未显式 pin seccomp/AppArmor。容器一旦逃逸（内核/runc CVE）即落在可写 docker.sock 旁。
- 修复：不要把 docker.sock 挂进可被网络访问的服务；用 rootless/sysbox 或 gVisor(`runsc`) 运行时；必须用 docker.sock 时前置 socket-proxy 限定到 `containers/create|start|exec|kill`，sidecar 跑非 root + userns-remap + 显式 seccomp/AppArmor。

**F2 — 默认无鉴权 → 未鉴权 RCE** 🔴
- 证据：`API_KEY=os.environ.get("SANDBOX_API_KEY","")`（`app.py:58`），中间件 `if API_KEY` 才校验（`:566-571`）→ 空 key 全端点免鉴权；compose 默认 `${SANDBOX_API_KEY:-}` 为空且监听 `0.0.0.0:48217`。忘设 key 即对外暴露未鉴权 `/exec`（任意 Python，紧邻 docker.sock）。即便设了也无 IP 白名单/mTLS。
- 修复：key 为空时 fail-closed（拒启或只绑 `127.0.0.1`）；只绑私网；网络策略仅允许主程序访问。
- 备注：本次部署已设非空 key，缓解了「默认空」的最坏情况，但默认值与 `0.0.0.0` 暴露仍应修。

**F5 — 无请求体大小限制 → OOM** 🟠：`/exec` 的 `code` 无长度上限（`app.py:299`），`/files` 的 20MB 检查在 Pydantic 已把整体读入内存**之后**；uvicorn 无 body 上限。修复：中间件按 `Content-Length` 先拒；`code` 加 1MB 上限。

**F6 — 无磁盘配额 + 默认根可写 → 磁盘 DoS** 🟠：`docker run` 无 `--storage-opt size=`，`READ_ONLY_ROOTFS` 默认 OFF，`/workspace` 为可写层；`open('/workspace/x','wb').write(b'0'*N)` 循环填满宿主磁盘（16 并发会话）。修复：默认 `--read-only` + tmpfs，设 `--storage-opt size`，限聚合磁盘。

**其他（MEDIUM/LOW）**
- 🟡 F3 web_search/image_generate 用 `http.DefaultClient` + admin base_url 无 SSRF 防护（`searcher.go:62`、`builtins.go:724`）→ 绕过 web_fetch 的硬化。修复：全部出站工具 HTTP 走 SSRF-safe 传输或域名白名单。
- 🟡 F4 web_fetch SSRF 漏 NAT64 `64:ff9b::/96`、CGNAT `100.64/10`、benchmark 段（`net_safety.go:13`）。注：十进制/八进制单数 IP（`http://2130706433/`）已被 `net.ParseIP` 返回 nil 而拦下。修复：补显式 CIDR 拒绝表。
- 🟡 F8 图片提示词黑名单为子串匹配（`builtins.go:627`），空格/leetspeak/同义词/Unicode 即绕过；分类合理但机制无效。修复：接真实 moderation API，先规范化再匹配，黑名单仅作预筛。
- 🟡 F9 `_safe_under_workspace` 仅 `normpath` 不解析符号链接（`app.py:526`），可 `symlink('/workspace/sl'→'/')` 后上传写出工作区（容器内）。修复：sidecar 内 `realpath -m` 复验前缀或 `O_NOFOLLOW`。
- 🟡 F10 `python_execute` 无每轮次数上限（`orchestrator.go:71` 无该工具条目），单消息可驱动大量 exec（与 F6 叠加成多租户 DoS）。修复：加每轮上限 + 按用户能力门控。
- 🟡 F11 会话 id 非每会话密钥（`app.py:359,452`）→ 鉴权关或 id 泄露时可跨租户 exec/删除。修复：会话绑定鉴权主体，session id 视为密钥不入日志。
- 🔵 web_fetch 未禁压缩（gzip 炸弹，受 25s 超时与 32K 截断限制）；🔵 artifact mime 来自沙盒并存储，下发若不带 `nosniff` 则 svg/html 可成存储型 XSS；🔵 `GetFile`/`/files/get` 死端点（404）；🔵 工作区 S3 归档凭据空传给忽略它的 sidecar。
- ✅ 已正确：web_fetch 的 SSRF 防护在**拨号时按解析 IP + 端口**复验（防 DNS rebinding，`net_safety.go:32`）；容器基本隔离参数（network=none、cap-drop ALL、no-new-privileges、非 root、pids/mem/cpu 限制）齐全；artifact 抽取有大小/数量上限。
