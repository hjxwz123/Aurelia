使用量と分析

「データ」メニューの下に、同じブックを見る2つの視点があります:「使用量」(/admin/usage) はい呼び出し記録会計、妨害及び清掃に用いられる「分析」(/admin/analytics) はい集合看板トレンドを見つけるために、大企業を見つけるためのデータソースは、使用量の記録表: すべての底層 モデル 呼び出し一行を記録し、ユーザーが見えるチャットの返信だけでなく、タイトル生成、クエリのルーティング、圧縮の概要、メモリ抽出、埋め込み、画像、審査、 ディプリサーチの計画と検証などの内部呼び出しも各行を記録します(それらは同じようにコストを生成し、ユーザーのコストに含まれるべきです)。

ワークスペース 所属​

シェア ワークスペース内の呼び出しは、「誰が支払うか」に従います: メンバーはシェア セッション中にチャットし、 クォータ と クレジット 消費記録送信者自分の名前で、量行で所属するスペースを同時に記録し、関係会話の下部に「スペース」ロゴを表示します。チームにコストを分割する際に、ユーザーのフィルタリングによって見られるのは、それぞれの人の実際の消費量であり、スペースロゴは個人の使用と協力の使用を区別するために使用されます。

記録を削除​

  • 内部ゴミ箱:単一のレコードを削除します。
  • 「フィルタの結果を削除」:現在のフィルタの命令のすべてのレコードを大量に削除し、ボトルウィンドウが命令の数を表示することを確認します。
削除は回数に影響します。

使用記録はレポートだけではありません:サービスが再起動した後、ユーザーのウィンドウ クォータの計算と時効 クレジットの消費はすべてこのテーブルから再播種されます。最近のクォータウィンドウ内にある記録は、ユーザーの「使用回数 / 使用回数 クレジット」を過小評価する可能性があります。 歴史アーカイブデータのクリーニングには影響がありませんので、以前の時間帯に基づいて削除することをお勧めします。

誤ったリクエストと失敗したロードログ​

ミスも入金。​

アップストリーム 呼び出しが失敗した場合(接続エラー、401 / 403 / 429 / 5xx など)**同じく一行。**すべてのトークン、コスト、クレジットがゼロで、エラー状態としてマークされ、失敗回数を統計し、「エラーのみ」でフィルタリングし、チャンネル品質の問題が一目瞭然です。

補充カルメットの保証:**誤差はユーザーに負担されません。**クォータウィンドウの回収、ユーザー側に見える使用量の数え方、分析パネルを除外し、失敗したリクエストはユーザーの無料回数を消費しず、「呼び出し数 / アクティブユーザー」をトークン / 料金と一致しない水分に浸透させない。

ポケットのロゴ​

モデルがチャンネルの底部を設定したとき、メインチャンネルが失敗すると、自動的にポケットの底部に転換して再試行します。ポケットの底部を横切るチャンネルにオレンジ色の「ポケットの底部」のロゴが表示されます(ツールサイクルのいずれかのリクエストでポケットの底部が切られ、全輪がマークされます)。

データ分析(データ分析)​

同一のログをトレンドビューにまとめ、すべての統計をエラーの排除呼び出し数、アクティブユーザー、トークン/コストは常に一致しています。

ウィンドウと指標​

  • 時間の窓: 右上隅に、最近の 1 / 7 / 30 / 90 日を選択します(スローアップウィンドウ、インターフェイス最大365日サポート)。
  • 指標変更: 呼び出し回数 / トークン / コスト、 1 つのスイッチはページ上のすべてのグラフを駆動します; トークン = 入力 + 出力の合計。

トレンド図​

柱状図に立って、ウィンドウ内の全体的な動きを表示する: 1 日 ウィンドウを選択するときに時刻バッテリー、残り天バッテリーを分割する; 停止する任意の柱は、その時段の正確な数値を示します。

トップモデル/トップユーザー​

2枚の並列カードがそれぞれ、窓内で最も消費したカードをリストします。前の8モデル与前 8 ユーザー(コスト順に順番に、コストは呼び出し量によって同じです):

  • 各行に一つの横向の比率線(現在の指標の最大値)、右側の数値。
  • トレンドグラフと共有するスパークラインを付加し、モデル/ユーザーの消費が継続的か突然であるかを一目で見る。
  • ユーザー バーはメールボックスを表示し、モデル バーはモデルの名前を表示します。

異常なコストの調査の典型的な経路:分析ページがユーザーのコストの急増を発見し、メールボックスを記録し、「使用量」ページに切って、そのユーザーにフィルタリングし、何のモデル、どのような用途、一度にいくつトークンを調べ、必要に応じて関連するセッションにアクセスしてコンテンツを確認します。

オンラインユーザー判断​

オンライン状態は「ユーザー > ユーザー」リストに表示されます:各行のメールボックスの前に、緑色がオンライン、グレーがオフラインでステータスポイントが表示されます。

  • ユーザー過去5分以内任意で認証されたリクエストは、オンラインとみなされます。
  • サーバーはリクエスト処理時にユーザーの最近のアクティブタイムを更新し、書き込み頻度を制御します。**毎分更新します。**したがって、オンラインでは最大約1分遅れが判定されます。
  • リクエストのタッチに完全に基づいて判断され、独立したハートチャンネルがない:ユーザーがページを開くが長い間何の操作も(メッセージを送らない、不要なページ)なしで徐々にオフラインに表示されることが期待される行動です。
分析タブレットと「アクティブユーザー」の区別

ユーザーリストの「オンライン」はこの瞬間ほぼリアルタイムの状態;分析タブレットのKPIの「アクティブユーザー」は統計ウィンドウ生成された モデル 呼び出された重量の数。 前者は「今誰がいるか」と答え、後者は「この期間に誰が使用しているか」と答えました。
